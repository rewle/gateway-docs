## Текущая реализация

Gateway In — не два сервиса, а часть одного процесса гейтвея (`src/backend/main.go: handleGateway`), той же, что обслуживает Gateway Out (`gateway_out.go`) и Gateway Peer (`gateway_peer.go`). Отдельного "аггрегатора источников" в системе нет — у каждого route один upstream (`routes.target`), без конфигурации множественных источников или схем подключения к ним; ретраи задаются на route (см. ниже), отдельной политики "на источник" нет.

### Поток запроса (`/gw/*`)

1. Определяется вызывающая сторона (`resolveConsumer`), способы проверяются в порядке peer → Bearer → Basic → сертификат; берётся первый, чьи креденшалы присутствуют в запросе, и если они неверны — сразу `401`, без перебора остальных способов:
   - доверенный подписанный запрос от другого инстанса гейтвея (Gateway Peer, см. [Gateway Peer](./gatewayPeer.md)) — доверенный peer получает доступ ко **всем** routes;
   - `Authorization: Bearer <token>` (`consumers.token`). Пустой токен отвергается явно (`401`) — иначе он совпал бы с консьюмером, у которого `token=''`, то есть открывал бы анонимный вход;
   - `Authorization: Basic base64(<consumer_id>:<token>)` — username это ID консьюмера, password — тот же `consumers.token`. Отдельного секрета нет, это второй транспорт того же токена для клиентов, которые умеют только Basic Auth;
   - клиентский сертификат (mTLS): консьюмер ищется по `consumers.cert_fingerprint` из заголовка `X-Client-Cert-Fingerprint`, если `Authorization` отсутствует; найден — аутентифицирован без токена (см. ниже).

   Сравнение токена в Bearer- и Basic-ветках — constant-time (`crypto/subtle`): Bearer-ветка перебирает всех консьюмеров без раннего выхода, а не ищет по токену через SQL-индекс.
2. По `path` + `method` ищется `route` (`findRoute`). Из всех подходящих по префиксу выбирается route с **самым длинным** `path` (`/a/b` выигрывает у `/a`), метод сравнивается без учёта регистра, `*` матчит любой. Порядок важен не только для маршрутизации: выбранный route определяет, какие правила маскирования применятся, поэтому он не должен зависеть от физического порядка строк в таблице.
3. Проверяется, что `route_ids` потребителя включает найденный route — иначе `403`.
4. Если у route задан `rate_limit` (> 0) — проверяется in-memory счётчик запросов на route за текущее часовое окно (`rateLimitExceeded`); при превышении — `429`, без обращения к upstream. Счётчик живёт в памяти процесса: не переживает рестарт и не общий между репликами (при нескольких инстансах фактический лимит умножается на их число).
5. После route-лимита проверяется **квота пары консьюмер+route** (`consumer_quotas`), если она заведена: отдельный лимит req/час для конкретного потребителя на конкретном route, тот же in-memory механизм и то же окно в час, превышение — тот же `429`. Оба лимита действуют одновременно. Квота ищется по фактическому `consumer_id`, поэтому применима и к доверенным peer'ам (`peer:<peer_id>`). Управление: `GET /api/quotas` (опционально `?consumer_id=`), `POST /api/quotas` (upsert по паре), `DELETE /api/quotas/{consumer_id}/{route_id}`; в админ-панели — секция Consumer quotas на странице Rules.
6. Запрос форвардится на `route.target`, вызов ограничен таймаутом `gatewayUpstreamTimeout` (30с) на весь цикл обработки, включая ретраи. Заголовок `Authorization` потребителя до upstream не доходит — он удаляется.
7. Ответ прогоняется через field rules (см. ниже).
8. Каждый запрос асинхронно логируется в ClickHouse (см. [Log Collector](./log.md)).

Схема ответа отдельно в админ-панели не задаётся — поля адресуются прямо по пути в `rules.field`, без отдельного шага сериализации по заранее сконфигурированной схеме.

### Ретраи к upstream

Если у route задан `retry_count` (0–5, поле Retries на странице Routes), неудачный вызов апстрима повторяется с паузой 200мс × номер попытки (`retryTransport` в `main.go`). Что считается ретраебельным:

- **dial-ошибка** (соединение не установилось, `net.OpError` c `Op=="dial"`) — ретраится для **любого** метода: запрос гарантированно не дошёл до апстрима, задвоения быть не может;
- **прочие транспортные ошибки** (обрыв после установления соединения) и **ответы 502/503/504** — только для идемпотентных методов (GET/HEAD/OPTIONS): POST апстрим мог успеть обработать до обрыва, повтор задвоил бы операцию.

Локальный `502 upstream response cannot be masked` (fail-closed маскирование, см. ниже) — **не** ошибка апстрима и не ретраится: `retryTransport` работает на уровне транспорта, ниже `ModifyResponse`, где эта ошибка рождается, поэтому исключение получается конструктивно, а не проверкой.

Тело запроса для реплея буферизуется в память (до 8 MiB — `maxProxyBodyBytes`); тело больше лимита не буферизуется — такой запрос уходит одной попыткой без ретраев. Весь цикл, включая паузы, живёт внутри общего таймаута `gatewayUpstreamTimeout` (30с).

### Аутентификация сертификатом (mTLS через reverse proxy)

TLS с клиентским сертификатом терминирует nginx (`ssl_verify_client optional_no_ca`), а не gateway. nginx кладёт отпечаток предъявленного сертификата в заголовок `X-Client-Cert-Fingerprint` (`$ssl_client_fingerprint`); gateway ищет консьюмера по `consumers.cert_fingerprint` — hex lowercase без двоеточий, при сохранении и при сравнении значение нормализуется (регистр, двоеточия).

- **Алгоритм — SHA-1**, потому что `$ssl_client_fingerprint` у nginx — SHA-1 и другого он не отдаёт. Это идентификатор конкретного сертификата, а не криптографическая защита самого канала: подлинность клиента доказывает TLS-handshake (владение приватным ключом), fingerprint лишь сопоставляет сертификат консьюмеру. Снять отпечаток: `openssl x509 -in client.crt -noout -fingerprint -sha1`.
- `optional_no_ca` — nginx проверяет владение ключом, но не цепочку CA: доверие определяется тем, что отпечаток заведён у консьюмера в админ-панели (по сути certificate pinning), self-signed сертификаты годятся.
- **Модель доверия — та же, что у `X-Real-IP`**: gateway верит заголовку безусловно, поэтому он достоверен только за собственным reverse proxy. nginx обязан перезаписывать входящий `X-Client-Cert-Fingerprint`, а не пропускать клиентский насквозь — в `src/nginx/nginx.conf` это делает `proxy_set_header X-Client-Cert-Fingerprint $ssl_client_fingerprint` (без сертификата переменная пуста, и nginx удаляет заголовок целиком). Если gateway доступен в обход nginx (dev/test-стек), заголовок может подставить кто угодно.
- **На текущем демо-стенде способ не работает**: внешний TLS терминирует Caddy (leadbot держит 80/443), и клиентские сертификаты до nginx не доходят. Способ рабочий только при прямом TLS-подключении к nginx.

Токен у консьюмера остаётся обязательным (для create/update), так что сертификат — дополнительный способ входа, а не замена токена.

### Field rules и маскирование

Правила берутся из таблицы `rules`, scoped по паре `consumer_id` + `route_id`. Действия — по dotted-path до поля, включая вложенные пути и массивы (`applyRules`/`applyRule`):

| Действие | Что делает |
|---|---|
| `hide` / `remove` | удаляет поле из ответа |
| `mask` | заменяет значение на `***` |
| `hash` | заменяет значение на HMAC-SHA256 от него (hex, 32 символа) |

`hash` — это **HMAC-SHA256 с солью** `GATEWAY_HASH_SALT`, а не голый sha256: несолёный хеш телефона, ИНН или email перебирается по словарю за секунды, то есть «хеширование» не скрывало бы значение. Соль общая для инстанса и обязана быть постоянной между рестартами — при её смене все ранее выданные партнёру псевдонимы полей изменятся, и сквозная аналитика по ним развалится. Если `GATEWAY_HASH_SALT` не задан, гейтвей поднимается со случайной эфемерной солью и пишет об этом в лог (годится только для дева).

### Маскирование fail-closed

Ключевая гарантия продукта: **если для консьюмера на этом route настроены field rules, он физически не может получить ответ, который гейтвей не смог замаскировать.** Раньше поведение было fail-open — неразобранное тело проходило насквозь сырым, то есть ровно те данные, ради сокрытия которых правило и заводилось, утекали при первом же нестандартном ответе upstream.

Сейчас (`ModifyResponse` в `handleGateway`) при непустом наборе правил ответ отдаётся партнёру, только если тело удалось разобрать и отфильтровать. Во всех остальных случаях — `502 upstream response cannot be masked` и запись в лог, а не сырое тело:

- `Content-Type` не содержит `application/json`;
- тело сжато (`Content-Encoding` не пустой и не `identity`);
- тело больше `maxProxyBodyBytes` (8 MiB) — лимит буферизации в память, без него один большой ответ upstream ронял бы процесс (контейнер живёт с `mem_limit 96m`);
- тело не парсится как JSON или результат не сериализуется обратно.

Чтобы это не срабатывало на каждом ответе, upstream'у при наличии правил принудительно отправляется `Accept-Encoding: identity` — сжатое тело замаскировать нельзя.

Если правил для пары консьюмер+route нет, тело не буферизуется и не разбирается вовсе — оно стримится как есть.

### Модель доверия peer: все routes, маскирование — per-peer правилами

Peer-запрос резолвится в синтетического консьюмера `Consumer{ID: "peer:<peer_id>"}` со списком **всех** route ID — доверенный peer по-прежнему видит все routes, ограничить ему набор routes нельзя. Но правила маскирования для него теперь заводятся как обычные `rules` с `consumer_id = "peer:<peer_id>"` (в выпадающем списке консьюмеров на странице Rules доверенные peer'ы показаны как `peer: <имя>`). Работает общий механизм: есть правила — маскирование применяется, **включая fail-closed** (немаскируемый ответ → `502`, а не сырое тело); нет правил — peer получает всё незамаскированным, как раньше (backward compatible: `trusted=true` без правил по-прежнему значит «отдавать этому гейтвею всё целиком»).

Коллизия `peer:<id>` с настоящим консьюмером исключена: id консьюмеров всегда генерирует сервер (`gen_random_uuid()`), клиентский id при создании игнорируется. Квоты (`consumer_quotas`) применимы к peer'ам тем же способом. Подробнее — в [Gateway Peer](./gatewayPeer.md).

## Чего нет относительно исходного замысла

- Нет отдельного сервиса-аггрегатора источников и связанной с ним конфигурации (строки подключения к БД, схемы API источника).
- Ретраи задаются на route (`retry_count`), отдельной политики "на источник" нет — источник у route один.
