## Текущая реализация

Gateway In — не два сервиса, а часть одного процесса гейтвея (`src/backend/main.go: handleGateway`), той же, что обслуживает Gateway Out (`gateway_out.go`) и Gateway Peer (`gateway_peer.go`). Отдельного "аггрегатора источников" в системе нет — у каждого route один upstream (`routes.target`), без конфигурации множественных источников, схем подключения к ним или политики ретраев.

### Поток запроса (`/gw/*`)

1. По `path` + `method` ищется `route` (`findRoute`).
2. Определяется вызывающая сторона:
   - либо по Bearer-токену (`consumers.token`), при условии что `route_ids` потребителя включает найденный route (`resolveConsumer`);
   - либо по доверенному подписанному запросу от другого инстанса гейтвея (Gateway Peer, см. `gateway_peer.go`) — тогда проверка по конкретному route не требуется, доверенный peer получает доступ ко всем routes.
3. Если у route задан `rate_limit` (> 0) — проверяется in-memory счётчик запросов на route за текущий часовой интервал (`rateLimitExceeded`); при превышении — `429`, без обращения к upstream. Счётчик живёт в памяти процесса и не переживает рестарт/масштабирование на несколько инстансов.
4. Запрос форвардится на `route.target`, вызов ограничен таймаутом `gatewayUpstreamTimeout` (30с) на весь цикл обработки.
5. Ответ (если JSON) прогоняется через `applyRules`/`applyRule` — правила `rules` (scoped по `consumer_id`+`route_id`) c действиями `hide`/`remove`/`hash`/`mask` по dotted-path до поля, включая вложенные пути и массивы.
6. Каждый запрос асинхронно логируется в ClickHouse (см. [Log Collector](./log.md)).

Схема ответа отдельно в админ-панели не задаётся — поля адресуются прямо по пути в `rules.field`, без отдельного шага сериализации по заранее сконфигурированной схеме.

## Чего нет относительно исходного замысла

- Нет отдельного сервиса-аггрегатора источников и связанной с ним конфигурации (строки подключения к БД, схемы API источника, политика ретраев).
