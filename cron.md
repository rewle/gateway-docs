### Cron Worker

## Текущая реализация

Cron Worker — не отдельный сервис, а часть процесса гейтвея (`src/backend/cron.go`), встроена через `robfig/cron/v3`. Job'ы хранятся в таблице `cron_jobs` и планируются in-memory при старте; CRUD и ручной запуск — через `/api/cron*`.

Планировщик держит состояние только в памяти процесса и рассчитан на единственный запущенный инстанс гейтвея — leader election между репликами отсутствует, при нескольких инстансах каждый выполнит job самостоятельно (дублирование запуска).

Алгоритм выполнения job'а (`executeJob`):

1. Воркер сам делает HTTP-запрос (`method`/`url`/`headers`/`body` из job'а) — без отдельного шага "идём в аггрегатор источников", аггрегатора в системе нет. Если у job'а задан `retry_count` (0–5, поле Retries в форме джобы), запрос к источнику повторяется при транспортной ошибке или ответе **≥500**, пауза 200мс × номер попытки; в `cron_runs` фиксируется финальный результат, при транспортной ошибке после ретраев в тексте ошибки указано число попыток (`(attempts N)`). Форвард (шаг 2) не ретраится. Весь прогон (запрос к источнику + ретраи + форвард) ограничен таймаутом **30 секунд**; ответ источника больше 8 MiB (`maxProxyBodyBytes`) не форвардится — прогон завершается ошибкой `response too large to forward`.
2. Ответ форвардится одним из двух способов:
   - если задан `outbound_route_id` — через соответствующий [Gateway Out](./gatewayOut.md) route: применяются field rules, инъекция заголовков, оба allow-листа (`source_ip_masks` и `dest_ip_masks`) и подпись peer-канала. Cron дёргает outbound-роут внутренним вызовом (`callOutboundRoute`), поэтому `source_ip_masks` проверяется с эффективным source-IP **`127.0.0.1`**: пустая маска — разрешено, непустая обязана содержать `127.0.0.1/32` (или покрывающий CIDR), иначе форвард завершится `403 source IP not allowed` и уйдёт в `last_error`/`cron_runs`. Запись в логах атрибутируется как `consumer_id = cron:<job_id>`.

     > **Breaking**: раньше cron-форвард `source_ip_masks` не проверял. Джобы, ходившие через роуты с непустой маской источника без loopback в списке, после обновления начнут получать отказ — в такие маски нужно добавить `127.0.0.1/32`;
   - иначе, если задан `forward_url` — ответ уходит `POST`-ом прямо на этот URL, **без** field rules, без allow-листов и без подписи. Это сырой форвард; для контролируемой отправки партнёру нужен `outbound_route_id`.
3. Статус (`last_run`/`last_status`/`last_error`) обновляется на самом job'е, и параллельно пишется полноценная запись в `cron_runs` (история прогонов, доступна через `GET /api/cron/{id}/runs`).
4. Ручной запуск без ожидания расписания — `POST /api/cron/{id}/run` (асинхронный: отвечает `202`, не дожидаясь прогона).

## Чего нет относительно исходного замысла

- Нет отдельного "аггрегатора источников" — воркер обращается к источнику напрямую по URL, заданному в job'е.
- Нет отдельного процесса/контейнера для cron — это часть gateway-бинаря, масштабируется и деплоится с ним вместе.
- Ретраится только запрос к источнику (`retry_count`); неудачный форвард не ретраится — ошибка фиксируется в `cron_runs`, следующая попытка будет только по расписанию (или вручную).
