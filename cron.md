### Cron Worker

> Диаграмма ниже относится к исходному замыслу (отдельный сервис + аггрегатор источников) и не отражает текущую реализацию.

## Текущая реализация

Cron Worker — не отдельный сервис, а часть процесса гейтвея (`src/backend/cron.go`), встроена через `robfig/cron/v3`. Job'ы хранятся в таблице `cron_jobs` и планируются in-memory при старте; CRUD и ручной запуск — через `/api/cron*`.

Планировщик держит состояние только в памяти процесса и рассчитан на единственный запущенный инстанс гейтвея — leader election между репликами отсутствует, при нескольких инстансах каждый выполнит job самостоятельно (дублирование запуска).

Алгоритм выполнения job'а (`executeJob`):

1. Воркер сам делает HTTP-запрос (`method`/`url`/`headers`/`body` из job'а) — без отдельного шага "идём в аггрегатор источников", аггрегатора в системе нет. Весь прогон (запрос к источнику + форвард) ограничен таймаутом **30 секунд**; ответ источника больше 8 MiB (`maxProxyBodyBytes`) не форвардится — прогон завершается ошибкой `response too large to forward`.
2. Ответ форвардится одним из двух способов:
   - если задан `outbound_route_id` — через соответствующий [Gateway Out](./gatewayOut.md) route: применяются field rules, инъекция заголовков, allow-лист **назначения** (`dest_ip_masks`) и подпись peer-канала. **`source_ip_masks` при этом НЕ проверяется** — cron дёргает outbound-роут внутренним вызовом (`callOutboundRoute`), в обход HTTP-входа, а маска источника проверяется только для HTTP-вызовов `POST /gw-out/{id}`. То есть ограничение по источнику не защищает от отправки, инициированной cron-джобой;
   - иначе, если задан `forward_url` — ответ уходит `POST`-ом прямо на этот URL, **без** field rules, без allow-листов и без подписи. Это сырой форвард; для контролируемой отправки партнёру нужен `outbound_route_id`.
3. Статус (`last_run`/`last_status`/`last_error`) обновляется на самом job'е, и параллельно пишется полноценная запись в `cron_runs` (история прогонов, доступна через `GET /api/cron/{id}/runs`).
4. Ручной запуск без ожидания расписания — `POST /api/cron/{id}/run` (асинхронный: отвечает `202`, не дожидаясь прогона).

## Чего нет относительно исходного замысла

- Нет отдельного "аггрегатора источников" — воркер обращается к источнику напрямую по URL, заданному в job'е.
- Нет отдельного процесса/контейнера для cron — это часть gateway-бинаря, масштабируется и деплоится с ним вместе.
- Нет ретраев неудачного прогона — ошибка фиксируется в `cron_runs`, следующая попытка будет только по расписанию (или вручную).
