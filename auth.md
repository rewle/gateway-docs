## Текущая реализация

Сервис авторизации (`src/services/auth`) — отдельный Go-процесс (свой `go.mod`, порт 8081) со своей БД PostgreSQL (`gateway_auth`). Он отвечает **только за аутентификацию пользователей админ-панели** (логин/сессия/SSO). Авторизация запросов партнёров к Gateway In/Out (проверка Bearer-токена потребителя, применение field rules) выполняется не здесь, а напрямую в гейтвее (`src/backend`) по таблицам `consumers`/`rules`/`routes` в основной БД `gateway` — отдельного высоконагруженного auth-узла, держащего все правила в памяти, в системе нет.

### Поток запроса

1. Пользователь логинится через `POST /auth/login` (или OAuth/LDAP-флоу) — сервис проверяет пароль (bcrypt, провайдер `LocalProvider`) либо делегирует проверку внешнему провайдеру.
2. При успехе сервис выпускает RS256 JWT (claims `sub`, `iss=gateway-docs-auth`, TTL 8 часов) и кладёт его в cookie `gw_session`.
3. Gateway (`backend/auth.go`) один раз при старте забирает публичный ключ с `GET /.well-known/jwks.json` и дальше валидирует JWT **локально**, без обращения к auth-сервису на каждый запрос — этим достигается требование быстрого ответа из исходного замысла, но не через кеш правил в памяти, а через статическую проверку подписи токена.
4. `/auth/*` (включая `/auth/me`, `/auth/logout`) проксируется гейтвеем на auth-сервис как есть.

### Провайдеры аутентификации (`AuthProvider`)

- **Local** — bcrypt-хеши в таблице `admin_users`, заводятся из env `ADMIN_USERS=user:secret,...` при старте (`seedUsers`).
- **OAuth 2.0** — authorization code flow (`OAUTH_CLIENT_ID/SECRET/AUTH_URL/TOKEN_URL/REDIRECT_URL`, опционально `OAUTH_USERINFO_URL`/`OAUTH_SCOPES`/`OAUTH_PROVIDER_NAME`); имя пользователя резолвится из userinfo-эндпоинта либо (без проверки подписи, только как имя, не как право доступа) из payload id_token.
- **LDAP** — включается через `LDAP_HOST`: бинд сервисным аккаунтом → поиск по `LDAP_USER_FILTER` → ребинд под найденным DN с паролем пользователя → имя берётся из `LDAP_USER_ATTR`. Пустой пароль отвергается до бинда (иначе LDAP-сервер может принять его как анонимный бинд).

#### `ADMIN_USERS`: хеш или плейнтекст

`secret` в `ADMIN_USERS=user:secret` — это либо **готовый bcrypt-хеш** (60 символов, префикс `$2a$`/`$2b$`/`$2y$`) — тогда он кладётся в `admin_users` как есть, либо **плейнтекст** — тогда он хешируется при старте, а в лог пишется предупреждение. Различение по префиксу принципиально: без него хеш попал бы в `bcrypt.GenerateFromPassword` повторно, и настоящим паролем стала бы сама строка хеша.

Для прода — только хеш, чтобы пароль не лежал в `.env` открытым:

```bash
htpasswd -bnBC 12 "" "mypassword" | tr -d ':\n'
```

**В `.env` знак `$` надо удваивать (`$$`)** — иначе docker compose интерполирует его как подстановку переменной и в контейнер приедет обрезанный хеш:

```
ADMIN_USERS=admin:$$2y$$12$$Ku0G...
```

#### Кто попадает в админку после OAuth (default-deny)

Успешная аутентификация у OAuth-провайдера **сама по себе не даёт права администрировать гейтвей** — иначе любой аккаунт в тенанте IdP получал бы полный доступ к настройкам обмена данными. `oauthUserAllowed` пускает пользователя, только если выполнено одно из:

- он уже заведён в `admin_users` (например, засидён через `ADMIN_USERS`);
- его имя перечислено в `OAUTH_ALLOWED_USERS` (список через запятую, сравнение регистронезависимое);
- оператор явно включил `OAUTH_AUTO_PROVISION=true` — тогда админ заводится автоматически для любого, кого аутентифицировал провайдер. Включать это можно, только если круг пользователей уже ограничен на стороне IdP.

Иначе — `403 user not provisioned for this gateway`.

#### Cookie сессии

`gw_session` — `HttpOnly`, `SameSite=Lax`, `Secure` (TTL 8 часов, как у самого JWT). `Secure` снимается только переменной `COOKIE_INSECURE=true` — она нужна локальному dev-стенду на `http://localhost`, где браузер иначе не примет куку, и не должна попадать в прод.

### Модели данных

```sql
admin_users (id, username, password_hash, provider)
oauth_states(state, expires_at)   -- CSRF-стейт OAuth-флоу, чистится фоновым job раз в 10 минут
```

### Чего нет относительно исходного замысла

- Нет protobuf-моделей `User`/`Rules`/`Limits`/`AuthRequest` и нет отдельного высоконагруженного эндпоинта авторизации партнёрских запросов — эту роль выполняет сам гейтвей.
- Нет проверки лицензии при старте/периодически.
- Нет учёта лимитов в реальном времени по логам и нет шины данных, которую слушал бы auth-сервис. Рейт-лимит применяется, но не здесь: `routes.rate_limit` проверяется самим гейтвеем (`rateLimitExceeded` в `backend/main.go`) по in-memory счётчику на route за текущее часовое окно, при превышении — `429` без обращения к upstream. Счётчик не переживает рестарт и не общий между репликами (см. [Gateway In](./gatewayIn.md)).
- Правила маскирования/хеширования живут не в auth-сервисе, а в таблице `rules` основной БД гейтвея и применяются гейтвеем синхронно на каждый запрос (см. [Gateway In](./gatewayIn.md)).

## Исходный замысел (историческая версия, не реализована)

Изначально предполагался отдельный высоконагруженный сервис авторизации, который при старте стягивает в память всех пользователей, все правила маскирования и все лимиты использования API, слушает шину логов для пересчёта остатков лимитов в реальном времени и периодически перезагружает лимиты из-за возможного внешнего сброса. Также предполагалась проверка подписанной лицензии (ограничения по числу партнёров и сроку действия) при старте и регулярно. Ниже — оценка памяти и модели данных из этого замысла, оставлены для истории:

Оценка памяти: при <200 партнёрств, <500 методов API и <200 типов полей на компанию — менее 20 млн правил по <200 байт, то есть <5 ГБ оперативной памяти.

```protobuf
message User {
    int32 user = 1;
    string authFunc = 2; // функция авторизации
    bytes authFuncData = 3; // данные по которым должна происходить сверка
}

message Rules {
    map<int32, UserRules> userRules = 1; // Ключ - пользователь
}
message UserRules {
    map<string, UserMethodRules> userMethodRules = 1; // Ключ - метод
}
message UserMethodRules {
    map<string, string> rule = 1; // Ключ - маска пути до поля данных, значение - скрытие или тип шифрования
}

message Limit {
    int32 user = 1;
    string method = 2;
    string type = 3; // Час, день, месяц и т.д.
    int32 count = 4;
}

message AuthRequest {
    bytes authData = 1;
    string authType = 2;
    string method = 3;
}
message AuthResponse {
    int32 code = 1;
    UserRules rules = 2;
}
```

Лицензия в этом замысле — данные об ограничениях системы, подписанные приватным ключом и проверяемые на старте и регулярно.
