# План рефакторинга

Составлен по итогам сквозного ревью 2026-07-12. Локальные недочёты уже исправлены
в том же ревью (fail-closed маскирование, whitelist по полному пути, OAuth
default-deny, солёный hash, XSS в админке, обход rate limit в leadbot,
`ADMIN_USERS` bcrypt/plaintext). Здесь — то, что не чинится точечной правкой.

Две части:

- **A. Дыры в модели безопасности** — код не делает того, что обещает продукт.
- **B. Паритет с whitepaper** — whitepaper обещает то, чего в коде нет вовсе.

Части независимы, порядок внутри каждой — по убыванию приоритета.

# A. Модель безопасности

## A1. Пир должен стать обычным консьюмером

**Что не так.** `resolveConsumer` (`backend/main.go:429`) для подписанного пира
возвращает синтетический `Consumer{ID: "gateway-peer"}` со списком всех
маршрутов. Правила затем грузятся запросом `WHERE consumer_id=$1 AND route_id=$2`
(`main.go` в `handleGateway`) — строки с `consumer_id='gateway-peer'` не бывает
никогда, потому что правила заводятся на реальные id из `consumers`. Итог:
`activeRules` пуст, маскирование не применяется, пир получает **все маршруты
сырыми**. Галочка `trusted=true` фактически означает «отдать этому гейтвею всё».

**Целевое поведение.** Пир — это консьюмер, который аутентифицируется подписью
вместо Bearer-токена. Всё остальное (скоуп маршрутов, правила, fail-closed
маскирование, rate limit, логи) работает для него ровно так же, как для партнёра.

**Шаги.**

1. Миграция в `migrate()` (`main.go:81`):
   ```sql
   ALTER TABLE consumers      ADD COLUMN IF NOT EXISTS kind       TEXT NOT NULL DEFAULT 'token';
   ALTER TABLE gateway_peers  ADD COLUMN IF NOT EXISTS consumer_id TEXT NOT NULL DEFAULT '';
   ```
   `consumers.kind` ∈ `token` | `peer`. Токен у пира пустой — поэтому проверку
   «пустой токен запрещён» (`createConsumer`/`updateConsumer`) применять только
   к `kind='token'`, иначе пира нельзя будет создать.

2. `trustPeer` (`gateway_peer.go:261`): при переводе в `trusted=true` — создать
   консьюмера `kind='peer'`, `name='peer: <url>'`, `route_ids='[]'` (ПУСТОЙ: маршруты
   выдаются явно) и записать его id в `gateway_peers.consumer_id`. При `trusted=false`
   — не удалять консьюмера, чтобы не терять настроенные правила; просто перестать
   его резолвить.

3. `verifyInboundPeer` (`gateway_peer.go:156`) возвращает не `bool`, а id
   консьюмера пира (или пусто). `resolveConsumer` по этому id грузит консьюмера
   из БД обычным запросом — синтетический `Consumer{ID:"gateway-peer"}` удалить
   полностью.

4. Frontend: на `peers.html` показать связанного консьюмера и ссылку на него; на
   `consumers.html` и `rules.html` пиры видны как обычные консьюмеры (`kind`
   отдавать в JSON, в UI помечать бейджем «peer», прятать поле токена).

**Ломающее изменение.** Сейчас доверенный пир автоматически получает все маршруты.
После фикса маршруты выдаются явно — иначе пир получит 403. Это надо описать в
`docs/gatewayPeer.md` и в release notes; при апгрейде существующих стендов —
одноразово проставить `route_ids` уже доверенным пирам (иначе интеграция встанет).

**Тесты.** Интеграционные: (а) пир получает 403 на маршрут, который ему не выдан;
(б) поле, закрытое правилом, замаскировано и для пира; (в) не-JSON ответ апстрима
при настроенных правилах даёт пиру 502, а не сырое тело.

## A2. Токены консьюмеров — хранить хеш, а не сам токен

**Что не так.** `consumers.token` лежит в БД открытым текстом, `GET /api/consumers`
возвращает его целиком, `consumers.html` кладёт полное значение в DOM (маска в
таблице косметическая). Любая XSS в админке = кража bearer-токенов всех партнёров
разом. Сравнение идёт `WHERE token=$1` — не constant-time.

**Важно: не bcrypt.** Токен — 128-битная случайная строка, а не человеческий
пароль, словарная атака на него неприменима. Медленный KDF на горячем пути прокси
недопустим (bcrypt cost 12 ≈ 100 мс на КАЖДЫЙ проксируемый запрос). Берём
HMAC-SHA256 с серверной солью — быстро, constant-time, от утечки дампа БД защищает.

**Шаги.**

1. Миграция:
   ```sql
   ALTER TABLE consumers ADD COLUMN IF NOT EXISTS token_hash   TEXT NOT NULL DEFAULT '';
   ALTER TABLE consumers ADD COLUMN IF NOT EXISTS token_prefix TEXT NOT NULL DEFAULT '';
   CREATE INDEX IF NOT EXISTS consumers_token_prefix_idx ON consumers (token_prefix);
   ```
2. Соль — переиспользовать `GATEWAY_HASH_SALT` (уже есть, `main.go:initHashSalt`)
   или завести `GATEWAY_TOKEN_SALT`. **Соль обязана быть постоянной**: при её смене
   все выданные токены перестанут проходить.
3. `createConsumer`: токен генерирует СЕРВЕР (`crypto/rand`, 32 байта), а не клиент.
   В ответе `POST /api/consumers` он отдаётся **единственный раз**. В БД пишем
   `token_hash` + `token_prefix` (первые 8 символов).
4. `resolveConsumer` (`main.go:429`): искать по `token_prefix`, затем
   `hmac.Equal(token_hash, hmac(предъявленный токен))`.
5. `listConsumers`/`updateConsumer`: поле `token` из JSON убрать совсем, отдавать
   `token_prefix`. Добавить `POST /api/consumers/{id}/rotate` — генерирует новый
   токен, показывает один раз.
6. Frontend `consumers.html`: показать токен в модалке после создания с кнопкой
   «скопировать» и явным предупреждением «больше не покажем»; из формы Edit поле
   токена убрать, вместо него кнопка «Rotate».

**Ломающее изменение.** Существующие открытые токены сохранить нельзя — в этом и
смысл. Миграция: одноразово перевыпустить токены и раздать партнёрам. Планировать
в окно.

## A3. Атрибуция исходящего трафика в логах

**Что не так.** `sendViaOutbound` (`gateway_out.go:211`) пишет в `log_entries`
только `route_id/path/status/duration_ms`; `consumer_id` и `method` пустые. По
логу нельзя ответить, кто инициировал отправку данных партнёру — админ через
`POST /gw-out/{id}` или cron-джоба. Для продукта, который продаёт аудит
(`whitepaper.md`), исходящий поток фактически не аудируется.

**Шаги.**

1. ClickHouse (`clickhouse.go:63`): `ALTER TABLE log_entries ADD COLUMN IF NOT
   EXISTS actor String DEFAULT ''`, плюс `actor_type LowCardinality(String)`
   (`admin` | `cron` | `peer` | `consumer`). Идемпотентный ALTER — как уже сделано
   для TTL.
   **Внимание:** `PrepareBatch(ctx, "INSERT INTO log_entries")` (`clickhouse.go:120`)
   вставляет по всем колонкам позиционно. При добавлении колонок ОБЯЗАТЕЛЬНО
   перевести на явный список: `INSERT INTO log_entries (ts, consumer_id, route_id,
   method, path, status, duration_ms, actor, actor_type)` — иначе `batch.Append`
   разъедется по арности и вставка сломается.
2. `LogEntry` (`main.go:50`) — поля `Actor`, `ActorType`.
3. `requireAuth` (`backend/auth.go:85`) уже валидирует JWT — положить `sub` в
   `context` (`context.WithValue`), чтобы `handleGatewayOut` знал username админа.
4. `sendViaOutbound` принимает актора параметром; `callOutboundRoute`
   (`gateway_out.go:287`) передаёт `actor_type='cron'`, `actor=<job id>`.
5. `queryLogs` (`clickhouse.go:172`) — добавить колонки в SELECT; `logs.html` —
   колонку «Инициатор».

## A4. Rate limit на `/auth/login`

Брутфорс пароля админа ничем не ограничен (`services/auth/main.go:80`): ни счётчика
попыток, ни задержки, ни блокировки.

Переиспользовать token bucket из `leadbot/ratelimit.go` (там уже рабочая реализация
с TTL-уборкой и потолком на размер карты) — скопировать в `services/auth`, ключ =
`IP + username`, лимит порядка 5/мин. Логировать превышения.

Заодно: `verifyInboundPeer` не защищён от replay — подписанный запрос можно
повторить в течение TTL (60 с), т.к. в JWT нет `jti`/nonce. Добавить `jti` и кэш
использованных id на время TTL (in-memory, как probeCache).

# B. Паритет с whitepaper

`docs/whitepaper.md` обещает четыре вещи, которых в коде нет. Это риск продаж, а не
техдолг: пока whitepaper уходит клиентам, каждое обещание — обязательство.

**По каждому пункту нужно решение: реализовать или переписать обещание.**
Рекомендация ниже — по соотношению цены и ценности.

## B1. Ретраи — РЕАЛИЗОВАТЬ (дёшево)

Обещано в § ADM Panel («ретраи на роуты/источники»), в коде нет нигде — ни в
Gateway In, ни в Gateway Out, ни в cron.

Схема: `ALTER TABLE routes ADD COLUMN retries INT NOT NULL DEFAULT 0`, то же в
`outbound_routes` и `cron_jobs` (+ `retry_backoff_ms INT NOT NULL DEFAULT 500`).

Реализация — только для **идемпотентных** случаев, иначе ретрай исходящей отправки
партнёру продублирует данные:
- Gateway Out (`sendViaOutbound`): ретраить только сетевые ошибки и 5xx, НЕ 4xx,
  с экспоненциальным backoff. Это самый ценный случай — партнёр моргнул, а мы не
  потеряли выгрузку.
- cron (`executeJob`, `cron.go`): ретраить и вызов источника, и форвард. Писать
  число попыток в `cron_runs`.
- Gateway In: ретрай проксирования GET/HEAD (идемпотентные). Для POST — не делать.

## B2. Квоты на потребителя — РЕАЛИЗОВАТЬ (дёшево)

Обещано; сейчас лимит только на маршрут (`routes.rate_limit`, `rateLimitExceeded`
в `main.go:692` — ключ по `routeID`).

Добавить `consumers.rate_limit INT NOT NULL DEFAULT 0` и второй счётчик с ключом
`consumerID` (или `consumerID+routeID`) в том же `sync.Map`. Проверять оба лимита,
срабатывает строгий. Это ~30 строк.

**Ограничение:** счётчик в памяти процесса → как и сейчас, корректен только для
одного инстанса gateway. Если нужны честные квоты на нескольких репликах — счётчик
надо выносить (Redis / Postgres), это отдельная задача.

## B3. Аутентификация консьюмера сертификатом или Basic Auth — ПЕРЕПИСАТЬ ОБЕЩАНИЕ

Обещано «сертификат, Basic Auth или токен» — есть только Bearer.

Basic Auth добавить дёшево (~20 строк в `resolveConsumer`), но он **не даёт ничего
поверх Bearer** и только расширяет поверхность атаки. mTLS полезен, но это не
правка в Go: нужен клиентский сертификат, его проверка на nginx
(`ssl_client_certificate` + `ssl_verify_client on`), проброс `X-Client-Cert-DN` в
gateway, доверие этому заголовку только от собственного прокси, плюс весь CA-обвяз
(выпуск, отзыв, ротация). Это отдельный проект.

Рекомендация: **убрать обещание из whitepaper**, оставив mTLS в роадмапе как
«по запросу». Если конкретный клиент требует mTLS — делать под него.

## B4. Ротация/выгрузка логов — РЕАЛИЗОВАТЬ ЧАСТИЧНО (дёшево)

Обещана настраиваемая ротация; в коде retention жёстко зашит (`TTL ts + INTERVAL
30 DAY`, `clickhouse.go:70`).

Минимум, закрывающий обещание: вынести срок в env `LOG_RETENTION_DAYS` (дефолт 30)
и применять `ALTER TABLE log_entries MODIFY TTL` на старте — механизм уже есть
(`clickhouse.go:79`), нужно только параметризовать. Плюс `GET /api/logs/export?format=csv`
для выгрузки.

Полноценную выгрузку во внешний сторадж (S3/SIEM) — в роадмап, не обещать сейчас.

## Что НЕ надо делать

- Возвращать fail-open маскирование «чтобы не ломать клиентов». Немаскируемый ответ
  апстрима при настроенных правилах — ошибка конфигурации, и 502 правильнее утечки.
- Хешировать токены консьюмеров bcrypt'ом (см. A2 — убьёт латентность прокси).
- Ослаблять whitelist Gateway Out обратно до верхнего уровня.
- Убирать `GATEWAY_HASH_SALT` ради «стабильности» — соль задаётся в `.env` и
  постоянна для инстанса, этого достаточно.
- Ретраить POST-запросы и исходящие отправки на 4xx (продублирует данные у партнёра).
