# План рефакторинга

Составлен по итогам сквозного ревью проекта 2026-07-12. Локальные недочёты уже
исправлены в том же ревью (fail-closed маскирование, whitelist по полному пути,
OAuth default-deny, солёный hash, XSS в админке, обход rate limit в leadbot,
`ADMIN_USERS` bcrypt/plaintext). Здесь — то, что не чинится точечной правкой и
требует решения по архитектуре.

Приоритет по убыванию.

---

## 1. Доверенный пир обходит маскирование целиком

**Проблема.** `resolveConsumer` (`backend/main.go`) для подписанного пира
возвращает синтетического `Consumer{ID: "gateway-peer"}` со списком всех
маршрутов. Правила затем грузятся запросом `WHERE consumer_id = 'gateway-peer'`
— такой строки в `rules` не бывает никогда, потому что правила заводятся на
реальные id консьюмеров из таблицы `consumers`. В результате `activeRules` пуст,
`ModifyResponse` не применяет ничего, и пир получает **все маршруты в
незамаскированном виде**.

Флаг `trusted=true` в админке фактически означает «отдать этому гейтвею всё, без
фильтрации». Это противоречит основной модели продукта (field-level masking на
контрагента) и не соответствует тому, что ожидает увидеть админ, ставящий
галочку.

**Решение.** Пир должен стать обычным консьюмером, а не исключением из модели:

- при `trustPeer` создавать (или связывать) строку в `consumers` — например
  `name = "peer: <url>"`, без токена, с явным `route_ids`;
- добавить `gateway_peers.consumer_id` → FK на `consumers.id`;
- `resolveConsumer` для пира возвращает этого реального консьюмера, дальше поток
  общий: те же `route_ids`, те же правила, то же fail-closed маскирование;
- в UI на странице Peers показать, какой консьюмер соответствует пиру, и увести
  настройку маршрутов/правил на существующие страницы.

**Побочный эффект, который надо не потерять:** сейчас доверенный пир получает
доступ ко всем маршрутам автоматически. После фикса маршруты придётся выдавать
явно — это изменение поведения, его нужно описать в `docs/gatewayPeer.md` и в
release notes.

**Тесты:** интеграционный тест «пир получает 403 на маршрут, который ему не
выдан» и «поле, закрытое правилом, замаскировано и для пира».

## 2. Токены консьюмеров хранятся и отдаются открытым текстом

**Проблема.** `consumers.token` лежит в БД в открытом виде, `GET /api/consumers`
возвращает его целиком, а `consumers.html` кладёт полное значение в DOM (маска
в таблице — чисто косметическая). Любая XSS в админке = кража bearer-токенов
всех партнёров разом. Сравнение токена идёт через `WHERE token = $1`, то есть
ещё и не constant-time.

**Решение.**

- хранить `token_hash` (bcrypt/argon2), а не сам токен;
- показывать токен ровно один раз — в ответе на `POST /api/consumers`; дальше в
  списке отдавать только префикс (`tok_a1b2…`) для опознания;
- аутентификация консьюмера: искать по префиксу, сверять хеш;
- миграция: одноразовая перегенерация токенов + предупреждение в release notes
  (сохранить существующие открытые токены нельзя — в этом и смысл).

Это ломает совместимость с уже выданными партнёрам токенами, поэтому — отдельной
задачей с осознанным окном.

## 3. Исходящий трафик неатрибутирован в логах

**Проблема.** `sendViaOutbound` пишет в `log_entries` только
`route_id/path/status/duration_ms`. `consumer_id` и `method` пустые, и нигде не
фиксируется, кто инициировал отправку — админ через `POST /gw-out/{id}` или
cron-джоба. Для продукта, который продаёт аудит (`whitepaper.md`), исходящий
поток фактически не аудируется: по логу нельзя ответить на вопрос «кто отправил
эти данные партнёру».

**Решение.**

- расширить `LogEntry` полем актора (`actor`, `actor_type`: `admin` / `cron` /
  `peer`) и колонкой в ClickHouse;
- `handleGatewayOut` берёт username из JWT (`requireAuth` сейчас валидирует
  токен, но не прокидывает `sub` дальше — нужно положить его в контекст);
- `callOutboundRoute` из cron проставляет id джобы;
- в `logs.html` показать колонку актора.

## 4. Нет rate limit на `/auth/login`

Брутфорс пароля админа ничем не ограничен. Логин идёт в auth-сервис, там нет ни
счётчика попыток, ни задержки, ни блокировки. Добавить per-IP + per-username
лимит (переиспользовать подход из `leadbot/ratelimit.go` — там уже есть рабочий
token bucket с TTL-уборкой).

---

## 5. Cron рассчитан на единственный инстанс

Состояние планировщика (`cronScheduler`, `cronEntries`) живёт в памяти процесса,
leader election нет. Несколько реплик gateway выполнят каждую джобу дублированно.
Сейчас это задокументированное ограничение и осознанный компромисс — но оно
блокирует горизонтальное масштабирование. Если оно понадобится: advisory-lock в
Postgres на время прогона джобы (`pg_try_advisory_lock(job_id)`) — самый дешёвый
способ, не требующий внешних зависимостей.

---

## 6. Дублирование в CRUD-хендлерах

`backend/main.go` — пять почти идентичных наборов list/create/update/delete,
отличающихся только SQL и типом. То же в `gateway_out.go` и `cron.go`. Это не
баг и не срочно, но объём кода на ровном месте втрое больше нужного, и каждая
новая сущность копирует ту же простыню (включая копирование ошибок вроде
проигнорированного `rows.Err()`).

Вынести generic-хелперы (`listHandler[T]`, `createHandler[T]`) на дженериках Go
1.22. Делать только вместе с одной из задач выше, чтобы не гонять диффы ради
диффов.

## Что НЕ надо делать

- Возвращать fail-open маскирование «чтобы не ломать клиентов». Если апстрим
  отдаёт не-JSON при настроенных правилах — это ошибка конфигурации, и 502
  правильнее утечки.
- Убирать `GATEWAY_HASH_SALT` ради стабильности хешей между стендами. Соль
  задаётся в `.env` и должна быть постоянной для инстанса — этого достаточно.
- Ослаблять whitelist Gateway Out обратно до верхнего уровня.
