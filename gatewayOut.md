### Gateway Out

Исходящий proxy: гейтвей отправляет данные наружу — партнёру, во внешнюю систему или в другой инстанс гейтвея. Реализован в `src/backend/gateway_out.go`, живёт в том же процессе, что Gateway In и Cron Worker.

В отличие от Gateway In, где гейтвей защищает **свои** данные от чужого запроса, здесь он ограничивает **свой же** исходящий трафик: кто изнутри контура имеет право инициировать отправку, куда эта отправка может уйти и какие поля в ней вообще допустимо вынести за периметр.

## Точка входа

```
POST /gw-out/{id}
```

`{id}` — идентификатор записи в таблице `outbound_routes`. Эндпоинт закрыт сессионным JWT (`requireAuth`), то есть дёргается изнутри контура — из админ-панели или внутренней системой, а не партнёром. Тело запроса (до 8 MiB, `maxProxyBodyBytes`) фильтруется по field rules и уходит на `target_url`.

Ответ — `{"status":"forwarded"}` со статусом, который вернул партнёр; тело ответа партнёра наружу не возвращается (оно вычитывается и отбрасывается).

Тот же путь исполнения переиспользует Cron Worker: job с заданным `outbound_route_id` форвардит ответ источника через outbound route (`callOutboundRoute`), минуя HTTP-слой. `source_ip_masks` при этом проверяется с эффективным source-IP `127.0.0.1` (см. ниже и [Cron Worker](./cron.md)).

## Модель данных

```sql
outbound_routes (
  id, name,
  target_url,                -- куда отправляем
  method,                    -- по умолчанию POST
  source_ip_masks   JSON,    -- кому можно инициировать отправку
  dest_ip_masks     JSON,    -- куда разрешено отправлять
  inject_headers    JSON,    -- заголовки, добавляемые к исходящему запросу
  timeout_s,                 -- таймаут запроса к партнёру, по умолчанию 30
  field_rules       JSON     -- allow/mask/remove по полям тела
)
```

CRUD — `GET/POST/PUT/DELETE /api/outbound`.

## Allow-листы по IP

Обе маски — списки IP или CIDR (`10.0.0.0/8`, `192.168.1.5`). **Пустой список означает «без ограничений»**, а не «запретить всё» — это осознанный дефолт, но при настройке боевого роута заполнять стоит оба.

- **`source_ip_masks`** — проверяется для всех инициаторов отправки. Для HTTP-вызовов `POST /gw-out/{id}` (`handleGatewayOut`) IP берётся из заголовка `X-Real-IP`, а при его отсутствии — из `RemoteAddr`. Значит, доверять этой маске можно, только если гейтвей стоит за собственным reverse proxy (`nginx/` в проде), который этот заголовок перезаписывает; если внешний клиент может дотянуться до гейтвея напрямую, он подделает `X-Real-IP` и обойдёт маску. Для внутренних вызовов из cron (`callOutboundRoute`) эффективный source-IP — **`127.0.0.1`**: чтобы cron-джобы ходили через роут с непустым `source_ip_masks`, в маски нужно добавить `127.0.0.1/32`.

  > **Breaking**: раньше `source_ip_masks` при вызове из cron не проверялся вовсе. Cron-джобы, форвардившие через роут с непустым `source_ip_masks` без `127.0.0.1/32` в списке, теперь получают `403 source IP not allowed`.
- **`dest_ip_masks`** — проверяется для всех отправок, включая cron: имя из `target_url` резолвится, и **каждый** полученный IP обязан попасть в allow-лист, иначе `403`.

### Защита от DNS-rebinding (`pinnedDialer`)

Проверить IP назначения и потом отдать URL в `http.Client` недостаточно: клиент выполнит **собственный** DNS-запрос, и между проверкой и соединением ответ резолвера может измениться (TOCTOU). Хост с коротким TTL, отвечающий сначала разрешённым адресом, а затем `169.254.169.254` или адресом внутренней сети, увёл бы туда исходящий трафик и заголовки, которые гейтвей инъектирует (в них обычно и лежат токены партнёра).

Поэтому при непустом `dest_ip_masks` транспорт клиента подменяется на `pinnedDialer`: он резолвит хост сам, валидирует по allow-листу **все** полученные адреса и подключается напрямую к уже проверенному IP, не давая клиенту резолвить хост заново. Окно между проверкой и соединением закрыто.

## Field rules (whitelist-семантика)

`field_rules` — список `{"field": "user.email", "action": "allow|mask|remove"}`, применяется к телу перед отправкой (`applyOutboundRules`). Семантика **другая, чем у Gateway In**, хотя названия действий похожи:

1. Если есть хотя бы одно правило `allow`, тело сначала прогоняется через whitelist (`keepFields`): всё, что не разрешено явно, из тела **вырезается**. Это защита от того, что источник однажды начнёт отдавать новое поле — новые поля не утекут партнёру просто потому, что их никто не запретил.
2. Поверх whitelist применяются `mask` (значение → `***`) и `remove` (поле удаляется).

Whitelist разрешает по **полному пути**, а не по верхнему ключу: `allow user.name` отдаёт партнёру только `user.name`, а не весь объект `user` целиком. Массивы фильтруются поэлементно — иначе whitelist не применялся бы вовсе к телу вида `[{...}]`.

Фильтрация тоже **fail-closed**: если правила настроены, а тело не разбирается как JSON или не сериализуется обратно, гейтвей возвращает `422` и **не отправляет** тело партнёру сырым.

Если правил нет, тело уходит как есть.

## Инъекция заголовков

`inject_headers` добавляются к исходящему запросу (`Content-Type: application/json` ставится всегда, до них). Это штатное место для ключей API партнёра — они лежат в конфиге роута, а не в теле вызывающей системы, поэтому внутренний потребитель может инициировать отправку, не зная секрета партнёра.

## Автоматический secure channel к другому гейтвею

Перед отправкой хост `target_url` пробится на `/.well-known/gateway` (`probePeerHost`, результат кешируется на 5 минут). Если на той стороне такой же гейтвей, запрос дополнительно подписывается и получает заголовки `X-Gateway-Auth` (RS256-JWT с хешем тела) и `X-Gateway-Origin` (`GATEWAY_ORIGIN_URL`) — принимающая сторона примет его как доверенный peer. Подробнее — [Gateway Peer](./gatewayPeer.md).

Подписывается тело **после** применения field rules — партнёр получает ровно то, что подписано.

## Логирование

Каждая отправка пишет запись в ClickHouse: `route_id` (id outbound-роута), `path` (`target_url`), `status`, `duration_ms`, `method` (метод исходящего запроса к партнёру, из конфига роута) и `consumer_id` — инициатор отправки:

- `ip:<адрес>` — HTTP-вызов `POST /gw-out/{id}` (адрес — `X-Real-IP` либо `RemoteAddr`);
- `cron:<job_id>` — форвард из cron-джобы.

Отказы (`403` по source/dest-маске, `502` при недоступном партнёре) логируются с той же атрибуцией. См. [Log Collector](./log.md).

## Ограничения

- Тело ответа партнёра не возвращается вызывающей стороне — только его HTTP-статус.
- Нет ретраев: неуспешная отправка — это `502` и запись в лог, повторить должен вызывающий (или cron по расписанию).
- Достоверность `source_ip_masks` держится на том, что `X-Real-IP` выставляет доверенный reverse proxy перед гейтвеем.
