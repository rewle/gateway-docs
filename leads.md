# Заявки с лендинга → Telegram (leadbot)

Статус: **в разработке** — сервис `leadbot` реализован и покрыт тестами (репозиторий `gateway-leadbot`, каталог `leadbot/` в воркспейсе), CI/CD-деплой на VPS настроен (GitHub Actions), не задеплоен фактически. Домен `gatewayb2b.ru` куплен, DNS A-запись `leads.gatewayb2b.ru` → IP VPS добавлена (проростает). Осталось: секреты деплоя в репозитории `gateway-leadbot`, `.env` на сервере, первый прогон CI, изменения в `landing/`.

## Задача и контекст

Форма «Запросить демо» на лендинге сейчас открывает mailto-черновик (`landing/script.js`). Нужно, чтобы заявка отправлялась автоматически и приходила основателю в Telegram.

Вводные:
- **Лендинг хостится на GitHub Pages** — статика, секретов на фронте быть не может; любой токен в JS публичен и как защита является фикцией.
- **Gateway на VPS — это демо-стенд**: возможность «поиграться» потенциальным клиентам и площадка для тестирования наших релизов. Его можно ронять, пересоздавать, чистить БД — продовый поток заявок к нему привязывать нельзя. nginx на сервере не запущен.
- **Код продукта не трогаем** — репозиторий `gateway` поставляется клиентам как отдельные инсталляции.

## Решение в двух словах

Отдельный автономный стек **leadbot** на том же VPS: два контейнера (Caddy для TLS + сам сервис, ~150 строк Go), собственный сабдомен, никакой связи с демо-стендом. Лендинг делает `fetch POST` напрямую в leadbot. Без токенов — эндпоинт публичный by design, антиспам на уровне самого сервиса.

```
браузер (лендинг на GitHub Pages)
  │  POST https://leads.<domain>/submit   (JSON, без auth)
  ▼
Caddy (контейнер, авто-Let's Encrypt, :80/:443)
  ▼
leadbot (контейнер)   CORS на origin лендинга, валидация,
  │                   honeypot, per-IP лимит, капы длин
  │  api.telegram.org sendMessage
  ▼
Telegram-бот → личный чат основателя
```

Демо-стенд gateway в этой схеме не участвует: заявки живут, даже когда стенд лежит или пересоздаётся под новый релиз. Прогон через Gateway In (рассматривался в первой версии дизайна) отклонён: consumer-токен в публичном JS — фейковая защита, а связывание продового потока заявок с демо-инсталляцией — лишняя хрупкость без выгоды.

## Компонент leadbot

Новый репозиторий `gateway-leadbot` (четвёртый, рядом с `gateway`/`gateway-docs`/`gateway-landing`; в воркспейсе — каталог `leadbot/`). Go stdlib, без фреймворков, без БД.

### API

`POST /submit` — JSON:

```json
{
  "name": "…",        // required, ≤200
  "company": "…",     // required, ≤200
  "email": "…",       // required, ≤200, наличие @ и точки в домене
  "phone": "…",       // optional, ≤50
  "message": "…",     // optional, ≤2000
  "website": ""       // honeypot: скрытое поле; непустое → 200 OK и молча дроп
}
```

Ответы: `200 {"ok":true}`, `400 {"error":"..."}` (валидация), `429` (per-IP лимит), `502` (Telegram недоступен после ретрая — фронт уходит в mailto-fallback).

`GET /healthz` — для healthcheck контейнера.

### Поведение

- CORS: `Access-Control-Allow-Origin` — только origin лендинга (GH Pages либо кастомный домен), preflight для `POST` + `Content-Type: application/json`. Заголовок Origin подделывается из curl, так что это гигиена, а не защита — защита ниже.
- `http.MaxBytesReader` 64 KB на тело.
- Per-IP token bucket в памяти (например 5/мин), IP — из `X-Forwarded-For`, который ставит Caddy.
- Honeypot `website` + капы длин полей.
- Telegram: `POST https://api.telegram.org/bot<TOKEN>/sendMessage`, `chat_id` из env, текст plain (без Markdown — нечего экранировать), timeout 10 c, один ретрай.
- Каждая принятая заявка дублируется в stdout (docker logs) — резервная копия на случай недоставки в Telegram.

Эндпоинт полностью публичный — худший сценарий абьюза: спам-сообщения в личку Telegram, срезаемый лимитами. Если станет проблемой — добавить Cloudflare Turnstile (бесплатен, работает со статики на GH Pages); в первую итерацию сознательно не берём.

### Конфигурация (env)

| Переменная | Назначение |
|---|---|
| `TELEGRAM_BOT_TOKEN` | токен бота от BotFather; живёт только на сервере |
| `TELEGRAM_CHAT_ID` | чат основателя (узнать через `getUpdates` после первого сообщения боту) |
| `ALLOWED_ORIGIN` | origin лендинга для CORS |
| `LISTEN_ADDR` | по умолчанию `:8090` |

### TLS и деплой

Caddy-контейнер с Caddyfile из двух строк:

```
leads.<domain> {
    reverse_proxy leadbot:8090
}
```

Let's Encrypt-сертификат Caddy получает и продлевает сам; порты 80/443 на VPS свободны (nginx не запущен). DNS A-запись `leads.gatewayb2b.ru` → IP VPS уже добавлена.

`docker-compose.yml` (dev, `build: .`) и `docker-compose.prod.yml` (прод, `image: ghcr.io/<repo>/leadbot`) в репозитории leadbot: сервисы `caddy` (публикует 80/443, volume под сертификаты) и `leadbot` (без публичных портов). Разворачивается на VPS в отдельном каталоге (например `~/leadbot/`), `.env` с телеграм-переменными — создаётся на сервере один раз вручную, CI его не создаёт и не трогает (токен бота никогда не попадает в GitHub Actions).

Деплой — через GitHub Actions (`.github/workflows/ci-cd.yml` в `gateway-leadbot`, по образцу `src/.github/workflows/ci-cd.yml`): тесты → сборка образа в `ghcr.io/<repo>/leadbot` → SCP конфигов (`docker-compose.prod.yml`, `Caddyfile`) и SSH-деплой на VPS секретами `DEPLOY_HOST/USER/SSH_KEY/PATH` того же репозитория (секреты между репозиториями не расшариваются, добавляются отдельно от `gateway`). В отличие от демо-стенда `gateway`, после деплоя сервис **не останавливается** — leadbot должен работать постоянно, в этом весь смысл («заявки живут, даже когда демо-стенд лежит»).

Бонус на будущее (вне скоупа): тот же Caddy умеет роутить по hostname — при желании `demo.<domain>` → `gateway:8080` даст демо-стенду валидный TLS без единого изменения в коде продукта.

## Изменения в landing

- `script.js`: вместо mailto — `fetch POST https://leads.<domain>/submit` с JSON-телом; URL константой вверху файла. Состояния UI: кнопка disabled на время отправки, успех — «Спасибо, вернёмся с датой демо», ошибка сети/5xx — fallback на текущий mailto-черновик (канал не теряется, даже если VPS лежит).
- `index.html`: скрытое honeypot-поле `website` (за пределами видимости, `tabindex=-1`, `autocomplete=off`); **обязательно** заменить footnote — текущий текст «ничего не отправляется автоматически и не уходит на сторонний сервер» станет неправдой. Новый честный вариант: «Заявка уходит напрямую нам, без сторонних форм-сервисов и аналитики».

## Отклонённые альтернативы

- **Через Gateway In демо-стенда** (первая версия дизайна) — consumer-токен на GH Pages публичен, т.е. фейковая защита; продовый поток заявок оказывается привязан к инсталляции, которую мы ломаем/пересоздаём при тестировании релизов.
- **Эндпоинт в backend gateway** — уедет клиентам в поставке.
- **Serverless (CF Worker) / форм-сервисы (Formspree и т.п.)** — внешняя зависимость и передача лидов третьей стороне при уже имеющемся VPS; лендинг обещает обратное.
- **Telegram API прямо из браузера** — светит bot token, неограниченный спам.
- **nginx + certbot вместо Caddy** — больше движущихся частей (два контейнера + cron продления) ради того же результата.
