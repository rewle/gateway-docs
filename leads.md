# Заявки с лендинга → Telegram (leadbot)

Статус: **реализовано и задеплоено** — сервис `leadbot` работает на VPS (`https://leads.gatewayb2b.ru`), форма на лендинге шлёт заявки, изменения в `landing/` запущены. Дополнительно: общий Caddy leadbot теперь служит единой точкой входа на 80/443 для всего VPS и проксирует ещё и demo-стенд `gateway` (домен `DEMO_DOMAIN`) через общую docker-сеть `edge` — оба сервиса могут работать на одном хосте одновременно, без конфликта портов.

## Задача и контекст

Форма «Запросить демо» на лендинге сейчас открывает mailto-черновик (`landing/script.js`). Нужно, чтобы заявка отправлялась автоматически и приходила основателю в Telegram.

Вводные:
- **Лендинг хостится на GitHub Pages** — статика, секретов на фронте быть не может; любой токен в JS публичен и как защита является фикцией.
- **Gateway на VPS — это демо-стенд**: возможность «поиграться» потенциальным клиентам и площадка для тестирования наших релизов. Его можно ронять, пересоздавать, чистить БД — продовый поток заявок к нему привязывать нельзя. Свой `nginx` демо-стенда host-порты не публикует (см. "Совместная работа с demo-стендом" ниже) — публикует их только Caddy leadbot.
- **Код продукта не трогаем** — репозиторий `gateway` поставляется клиентам как отдельные инсталляции.

## Решение в двух словах

Отдельный автономный стек **leadbot** на том же VPS: два контейнера (Caddy для TLS + сам сервис, ~150 строк Go), собственный сабдомен, никакой связи с демо-стендом. Лендинг делает `fetch POST` напрямую в leadbot. Без токенов — эндпоинт публичный by design, антиспам на уровне самого сервиса.

```
браузер (лендинг на GitHub Pages)
  │  POST https://leads.<domain>/submit   (JSON, без auth)
  ▼
Caddy (контейнер, авто-Let's Encrypt, :80/:443)
  ▼
leadbot (контейнер)   CORS на origin лендинга, валидация,
  │                   honeypot, per-IP лимит, капы длин
  │  api.telegram.org sendMessage
  ▼
Telegram-бот → личный чат основателя
```

Демо-стенд gateway в этой схеме не участвует: заявки живут, даже когда стенд лежит или пересоздаётся под новый релиз. Прогон через Gateway In (рассматривался в первой версии дизайна) отклонён: consumer-токен в публичном JS — фейковая защита, а связывание продового потока заявок с демо-инсталляцией — лишняя хрупкость без выгоды.

## Компонент leadbot

Новый репозиторий `gateway-leadbot` (четвёртый, рядом с `gateway`/`gateway-docs`/`gateway-landing`; в воркспейсе — каталог `leadbot/`). Go stdlib, без фреймворков, без БД.

### API

`POST /submit` — JSON:

```json
{
  "name": "…",           // required, ≤200 — как обращаться
  "contactMethod": "…",  // required, "email" | "phone" | "telegram"
  "contact": "…",        // required, ≤200; при contactMethod=email — наличие @ и точки в домене
  "website": ""          // honeypot: скрытое поле; непустое → 200 OK и молча дроп
}
```

Форма изначально спрашивала имя/компанию/email/телефон/сообщение — по фидбеку после первого демо укорочена до трёх полей (имя, способ связи, контакт), чтобы не отпугивать длиной.

Ответы: `200 {"ok":true}`, `400 {"error":"..."}` (валидация), `429` (per-IP лимит), `502` (Telegram недоступен после ретрая — фронт уходит в mailto-fallback).

`GET /healthz` — для healthcheck контейнера.

### Поведение

- CORS: `Access-Control-Allow-Origin` — только origin лендинга (GH Pages либо кастомный домен), preflight для `POST` + `Content-Type: application/json`. Заголовок Origin подделывается из curl, так что это гигиена, а не защита — защита ниже.
- `http.MaxBytesReader` 64 KB на тело.
- Per-IP token bucket в памяти (например 5/мин), IP — из `X-Forwarded-For`, который ставит Caddy.
- Honeypot `website` + капы длин полей.
- Telegram: `POST https://api.telegram.org/bot<TOKEN>/sendMessage`, `chat_id` из env, текст plain (без Markdown — нечего экранировать), timeout 10 c, один ретрай.
- Каждая принятая заявка дублируется в stdout (docker logs) — резервная копия на случай недоставки в Telegram.

Эндпоинт полностью публичный — худший сценарий абьюза: спам-сообщения в личку Telegram, срезаемый лимитами. Если станет проблемой — добавить Cloudflare Turnstile (бесплатен, работает со статики на GH Pages); в первую итерацию сознательно не берём.

### Конфигурация (env)

| Переменная | Назначение |
|---|---|
| `TELEGRAM_BOT_TOKEN` | токен бота от BotFather; живёт только на сервере |
| `TELEGRAM_CHAT_ID` | чат основателя (узнать через `getUpdates` после первого сообщения боту) |
| `ALLOWED_ORIGIN` | origin лендинга для CORS |
| `LISTEN_ADDR` | по умолчанию `:8090` |

### TLS и деплой

Caddy-контейнер с Caddyfile из двух строк:

```
leads.<domain> {
    reverse_proxy leadbot:8090
}
```

Let's Encrypt-сертификат Caddy получает и продлевает сам. DNS A-запись `leads.gatewayb2b.ru` → IP VPS уже добавлена.

`docker-compose.yml` (dev, `build: .`) и `docker-compose.prod.yml` (прод, `image: ghcr.io/<repo>/leadbot`) в репозитории leadbot: сервисы `caddy` (публикует 80/443, volume под сертификаты) и `leadbot` (без публичных портов). Разворачивается на VPS в отдельном каталоге (например `~/leadbot/`).

Деплой — через GitHub Actions (`.github/workflows/ci-cd.yml` в `gateway-leadbot`, по образцу `src/.github/workflows/ci-cd.yml`): тесты → сборка образа в `ghcr.io/<repo>/leadbot` → SCP конфигов (`docker-compose.prod.yml`, `Caddyfile`) → SSH-деплой на VPS. `.env` на сервере генерирует сам CI-шаг деплоя (`printf` из секретов Actions в SSH-сессии, `chmod 600`) — не хранится в репозитории и не создаётся руками. Все нужные значения — секреты репозитория `gateway-leadbot` (отдельно от `gateway` — секреты между репозиториями не расшариваются): `DEPLOY_HOST/USER/SSH_KEY/PATH` для SSH и `LEADS_DOMAIN`, `TELEGRAM_BOT_TOKEN`, `TELEGRAM_CHAT_ID`, `ALLOWED_ORIGIN`, `DEMO_DOMAIN` для `.env`. Каталог `DEPLOY_PATH` на сервере должен существовать заранее — это единственный ручной шаг. В отличие от демо-стенда `gateway`, после деплоя сервис **не останавливается** — leadbot должен работать постоянно, в этом весь смысл («заявки живут, даже когда демо-стенд лежит»).

### Совместная работа с demo-стендом gateway (общий VPS, порты 80/443)

Изначально предполагалось (см. «бонус на будущее» — теперь реализован), что тот же Caddy умеет роутить по hostname на demo-стенд без единого изменения в коде продукта. По факту понадобилась одна правка инфраструктуры на стороне `gateway`: его `nginx` (репозиторий `gateway`, `docker-compose.prod.yml`) перестал публиковать `80:80`/`443:443` на host — иначе он конфликтовал с уже занятыми Caddy leadbot портами при попытке поднять оба стека разом.

Схема: `nginx` demo-стенда и `caddy` leadbot подключены к общей внешней docker-сети `edge` (создаётся идемпотентно `docker network create edge` в обоих CI-деплоях). Второй site-блок в `Caddyfile` leadbot:

```
{$DEMO_DOMAIN} {
    reverse_proxy https://nginx:443 {
        transport http {
            tls_insecure_skip_verify
        }
    }
}
```

Caddy получает для `$DEMO_DOMAIN` валидный Let's Encrypt-сертификат снаружи; до `nginx` demo-стенда внутри сети `edge` — тот же self-signed TLS, что и раньше (внутренний хоп, `tls_insecure_skip_verify` оправдан — трафик не покидает docker-сеть на одном хосте). `nginx` при этом продолжает сам добавлять security-заголовки и проксировать на `gateway:8080`, как и раньше — изменился только способ, которым к нему приходит внешний трафик.

Требуется секрет `DEMO_DOMAIN` в `gateway-leadbot` (например `demo.gatewayb2b.ru`) и соответствующая DNS A-запись → IP VPS. CI `gateway` тоже создаёт сеть `edge` идемпотентно (на случай, если задеплоен раньше leadbot), и его smoke-тест теперь ходит через `docker compose exec nginx wget ...` изнутри контейнера — прямого host-порта 443 у него больше нет.

Благодаря этому leadbot (постоянно) и demo-стенд `gateway` (по требованию, для показа клиенту) могут быть подняты одновременно — заявки продолжают приходить, пока идёт демонстрация.

## Изменения в landing

- `script.js`: вместо mailto — `fetch POST https://leads.<domain>/submit` с JSON-телом; URL константой вверху файла. Состояния UI: кнопка disabled на время отправки, ошибка сети/5xx — красный текст под формой + fallback на mailto-черновик (канал не теряется, даже если VPS лежит).
- `index.html`: скрытое honeypot-поле `website` (за пределами видимости, `tabindex=-1`, `autocomplete=off`); footnote «Заявка уходит напрямую нам, без сторонних форм-сервисов и аналитики» вместо неактуального «ничего не отправляется автоматически».
- **Успех — не текст под кнопкой, а замена формы.** Изначальная реализация (маленький текст «Спасибо…» под кнопкой + `form.reset()`) была неочевидна — форма визуально возвращалась в пустое состояние, будто ничего не произошло. Заменено на отдельный блок `#contact-form-success` (скрыт по умолчанию, `hidden`): при успешном `200` форма прячется (`hidden = true`) и на её месте показывается заметная панель «✓ Заявка отправлена». Одновременно ставится cookie `gw_lead_sent` (365 дней) — при повторном заходе с того же браузера форма не показывается вовсе, сразу видна success-панель (проверка `document.cookie` при загрузке скрипта). Cookie ставится только на подтверждённый `200 ok` от leadbot — при падении в mailto-fallback не ставится, форма остаётся доступной для повторной попытки.
- **Поля формы укорочены** (см. «API» выше): вместо имя/компания/email/телефон/сообщение — три поля: «Как обращаться» (имя), «Способ связи» (`<select>` email/telegram/телефон) и «Контакт» (один input, у которого `type`/`autocomplete`/`placeholder`/подпись переключаются JS-обработчиком на `change` селекта — email → `type=email`, telegram → `type=text` с плейсхолдером `@username`, телефон → `type=tel`).

## Отклонённые альтернативы

- **Через Gateway In демо-стенда** (первая версия дизайна) — consumer-токен на GH Pages публичен, т.е. фейковая защита; продовый поток заявок оказывается привязан к инсталляции, которую мы ломаем/пересоздаём при тестировании релизов.
- **Эндпоинт в backend gateway** — уедет клиентам в поставке.
- **Serverless (CF Worker) / форм-сервисы (Formspree и т.п.)** — внешняя зависимость и передача лидов третьей стороне при уже имеющемся VPS; лендинг обещает обратное.
- **Telegram API прямо из браузера** — светит bot token, неограниченный спам.
- **nginx + certbot вместо Caddy** — больше движущихся частей (два контейнера + cron продления) ради того же результата.
